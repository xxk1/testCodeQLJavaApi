## 产品概述

针对 `/v1/courseTables/studentCourseList` 接口进行性能优化，该接口用于查询学生的课程列表信息。

## 核心问题

- **吞吐量低**：仅 17.3 次/秒
- **响应时间长**：平均 5289ms（超过 2 秒阈值）
- **CPU 高负载**：接口服务器 CPU 使用率达 97%（数据库 CPU 仅 16%，说明瓶颈在应用层）
- **网络带宽瓶颈**：1098.95Mb/s（超过 1000Mbps 限制）

## 性能问题根因分析

### CPU 高负载原因（应用层计算密集）

1. **EAGER Fetch 级联加载过度**：

- `CourseTable.courseTableDetailList` 使用 `FetchType.EAGER`
- `CourseTableDetail.courseTableDetailTeacherList` 使用 `FetchType.EAGER`
- `Group.courseTableList` 使用 `FetchType.EAGER`
- 导致每次查询加载大量不需要的关联数据，并触发多次数据库查询

2. **深度嵌套 Stream 操作**：

- `getTeacherIdList` 方法创建多个中间集合，进行 3 次 Stream 遍历
- 每次遍历都创建新的 List 对象，内存分配开销大

3. **重复计算与低效循环**：

- `getCourseStudentResources` 方法对每门课程重复调用辅助方法
- `GroupMemberList` 每次都要遍历过滤计算班级人数

### 网络带宽瓶颈原因

1. **数据传输量过大**：

- EAGER 加载了大量不需要的关联数据（CourseTableDetail、CourseTableDetailTeacher）
- 响应数据包含完整的对象关系图

2. **序列化数据冗余**：

- Jackson 序列化大量嵌套对象，增加 CPU 和网络开销

## 优化目标

- 响应时间降低至 500ms 以内
- CPU 使用率降低至 50% 以下
- 网络带宽占用降低至 500Mbps 以下
- 吞吐量提升至 100 次/秒以上

## 技术栈

- **框架**：Spring Boot 2.7.6 + Spring Data JPA
- **数据库**：MySQL
- **序列化**：Jackson
- **构建工具**：Gradle

## 实现方案

### 方案一：实体模型优化（推荐优先实施）

**问题**：`FetchType.EAGER` 导致无条件加载所有关联数据

**优化措施**：

1. 将 `CourseTable.courseTableDetailList` 改为 `FetchType.LAZY`
2. 将 `CourseTableDetail.courseTableDetailTeacherList` 改为 `FetchType.LAZY`
3. 将 `Group.courseTableList` 改为 `FetchType.LAZY`
4. 使用 `@EntityGraph` 或 JPQL JOIN FETCH 按需加载

**预期效果**：

- 数据加载量减少 60%-80%
- 数据库查询次数减少
- 内存占用显著降低

### 方案二：查询优化

**问题**：Specification 查询未使用 JOIN FETCH，导致 N+1 问题

**优化措施**：

1. 在 `CourseTableSpecification.getStudentCourseList` 中添加 FETCH JOIN
2. 添加数据库索引（group_member.student_id、group_member.open_id、course_table.school_year、course_table.term）
3. 使用 `query.distinct(true)` 避免重复数据

**关键代码修改位置**：

- `CourseTableSpecification.java` 第 166-183 行

### 方案三：代码层优化

**问题**：Stream 操作低效，创建大量中间集合

**优化措施**：

1. **合并 Stream 操作**：将 `getTeacherIdList` 中的多个 Stream 合并为单次遍历
2. **使用 Set 去重**：直接使用 HashSet 收集，避免 `distinct()` 遍历
3. **预计算缓存**：对同一 CourseTable 列表只计算一次 TeacherId/TeacherName
4. **简化 GroupMember 计算**：使用数据库 COUNT 查询替代内存遍历

**关键代码修改位置**：

- `CourseTableService.java` 第 155-161 行
- `CourseTableService.java` 第 285-327 行

### 方案四：架构层优化

**问题**：无缓存机制，每次请求都进行完整计算

**优化措施**：

1. **添加 Redis 缓存**：缓存学生课程列表（TTL 5-10 分钟）
2. **添加本地缓存**：使用 Guava Cache 缓存热点数据
3. **添加分页支持**：减少单次返回数据量

## 性能优化实现细节

### 数据流优化前后对比

**优化前**：

```
请求 → 查询所有 CourseTable → EAGER 加载所有关联 → 内存处理 → 序列化大对象 → 返回
```

**优化后**：

```
请求 → 检查缓存 → JOIN FETCH 精确查询 → 轻量处理 → 序列化精简对象 → 返回
```

### 优化后的 Stream 操作示例

```java
// CourseTableService.java - 优化后的 getTeacherIdList
private List<String> getTeacherIdList(List<CourseTable> courseTables) {
    Set<String> teacherIds = new HashSet<>();
    for (CourseTable ct : courseTables) {
        if (ct.getCourseTableDetailList() != null) {
            for (CourseTableDetail ctd : ct.getCourseTableDetailList()) {
                if (ctd.getCourseTableDetailTeacherList() != null) {
                    for (CourseTableDetailTeacher teacher : ctd.getCourseTableDetailTeacherList()) {
                        teacherIds.add(teacher.getTeacherId());
                    }
                }
            }
        }
    }
    return new ArrayList<>(teacherIds);
}
```

## 注意事项

1. **实体修改影响范围**：将 EAGER 改为 LAZY 可能影响其他调用处，需全局测试
2. **缓存一致性**：课程数据变更时需清除对应缓存
3. **事务边界**：确保 LAZY 加载在事务范围内执行
