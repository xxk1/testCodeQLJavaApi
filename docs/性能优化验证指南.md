# 学生课程列表接口性能优化验证指南

## 📋 优化概览

### 优化前后对比

| 指标 | v1 版本 | v2 版本（预期） | 提升幅度 |
|-----|--------|---------------|---------|
| **响应时间** | 5289ms | < 300ms | ↓ 94% |
| **CPU 使用率** | 97% | < 20% | ↓ 79% |
| **网络带宽** | 1098Mb/s | < 200Mb/s | ↓ 82% |
| **吞吐量** | 17.3次/秒 | > 150次/秒 | ↑ 767% |
| **数据库查询次数** | 211 次 | 2-3 次 | ↓ 98% |
| **表关联数量** | 5 张表 | 4 张表 | ↓ 20% |

### 核心优化点

1. ✅ 使用 `EntityManager.createNativeQuery` 动态查询
2. ✅ 利用冗余字段减少部分表关联（course_name, college_name）
3. ✅ 关联 `tb_course` 表获取 `course_code` 字段（必需字段）
4. ✅ 教师信息从 `tb_course_table_detail_teacher` 表单独查询，确保数据准确性
5. ✅ 精确查询必需字段，避免 `SELECT *`
6. ✅ 批量查询替代 N+1 查询（班级人数、教师信息）
7. ✅ 独立 v2 接口，零影响 v1

### 6. 性能基准标准

#### 6.1 响应时间标准

- ✅ 优秀：< 100ms
- ✅ 良好：100ms - 300ms
- ⚠️ 需优化：300ms - 1000ms
- ❌ 不可接受：> 1000ms

#### 6.2 吞吐量标准

- ✅ 优秀：> 300 次/秒
- ✅ 良好：100 - 300 次/秒
- ⚠️ 需优化：50 - 100 次/秒
- ❌ 不可接受：< 50 次/秒

#### 6.3 资源消耗标准

- ✅ 优秀：CPU < 20%, 内存增长 < 10%
- ✅ 良好：CPU 20%-40%, 内存增长 < 20%
- ⚠️ 需优化：CPU 40%-70%, 内存增长 < 50%
- ❌ 不可接受：CPU > 70%, 内存增长 > 50%

### 7. 问题排查指南

#### 7.1 如果 v2 响应时间未达预期

**检查项**：
1. ✅ 索引是否成功创建（包括教师信息查询相关索引）
2. ✅ 数据库统计信息是否更新（ANALYZE TABLE）
3. ✅ 冗余字段是否填充完整（course_name, college_name）
4. ✅ 教师信息表是否有数据（tb_course_table_detail_teacher）
5. ✅ 网络延迟是否正常
6. ✅ JVM 内存是否足够

**优化建议**：
```sql
-- 更新表统计信息
ANALYZE TABLE tb_group_member;
ANALYZE TABLE tb_course_table;
ANALYZE TABLE tb_group;
ANALYZE TABLE tb_course_table_detail;
ANALYZE TABLE tb_course_table_detail_teacher;

-- 查看索引使用情况
SHOW INDEX FROM tb_group_member;
SHOW INDEX FROM tb_course_table_detail;
SHOW INDEX FROM tb_course_table_detail_teacher;

-- 验证教师信息是否存在
SELECT COUNT(*) FROM tb_course_table_detail_teacher;
```

#### 7.2 如果 CPU 使用率仍然很高

**检查项**：
1. ✅ 是否有其他高 CPU 进程
2. ✅ JVM GC 频率是否正常
3. ✅ 数据库连接池配置是否合理

**优化建议**：
```bash
# 查看 JVM GC 情况
jstat -gcutil <pid> 1000 10

# 查看 JVM 线程情况
jstack <pid> > thread_dump.txt
```
